# 00-总体处理规则

> 本工程的最高规则。所有处理动作必须遵守本文件，冲突时以本文件为准。

## 1. 不可违反的硬约束

1. **不修改旧目录**：原内容根目录只读。所有处理只发生在本工程（obsidianvault/）内。
2. **原始素材先复制**：纳入范围的素材必须先复制到 `01-原始素材区/完整副本/`，并记录 SHA256，处理永远基于副本。
3. **先分类，后处理**：任何来源必须先进入 [[来源候选清单]] 并完成分类，才能决定三种去向之一：
   - **跳过**：无知识价值（空文件、纯配置、占位符）→ 记录原因，不进入工程正文。
   - **归一化**：本身是结构件/导航件（主页、旧 MOC）→ 内容并入对应 MOC 或项目页，不单独成为内容单元。
   - **抽取**：含可迁移知识 → 拆解为内容单元。
4. **样本先行**：先用代表性样本验证结构，结构稳定（脚本验证通过）后才允许批量推进。
5. **禁止孤立文件**：每个正文笔记必须被至少一个 MOC 或索引页通过 `[[ ]]` 引用，且自身至少有 1 条出链。由 `_system/脚本层/validate_vault.py` 强制校验。

## 2. 处理流水线（每条来源走一遍）

```
来源发现 → 登记[[来源候选清单]] → 登记[[原始索引]]（含哈希）
  → 分类（跳过/归一化/抽取）→ 进入[[待处理清单]]
  → 抽取内容单元（见 01-内容单元规则）
  → 建立显式关系（见 02-关系规则）
  → 挂靠 MOC（见 04-目录与挂靠规则）
  → 登记[[去重候选]] → 更新[[处理状态总览]]
  → 运行 validate_vault.py，必须全绿
```

## 3. 优先级

P1 = 本人已发布内容 → P2 = 本人未发布但较成熟稿件 → P3 = 外部研究素材。
同批次内按 P1→P2→P3 顺序处理，优先级写入来源候选清单。

## 4. 引用规范

正文中所有 markdown 引用（`[文字](路径.md)` 或裸路径引用）统一改写为 Obsidian 直链 `[[文件名]]`。外部网址保留 URL 格式。原始素材区的副本**不做改写**，保持逐字节原样。
